- 纯 Python 标准库，**零外部依赖**

- 无框架、无构建步骤、无运行时依赖

## 二、商业物业场景的关键缺口

### P0 — 必须补齐才能用于商业物业

| # | 缺失功能 | 说明 | 估算工作量 |
|---|---------|------|-----------|
| 1 | **公摊费用按面积/用量比例分摊** | 当前只有按户固定金额。商业物业需要按建筑面积比例、或按用量比例分摊水电公摊 | 3-5天 |
| 2 | **合同驱动计费** | 合同期限、单价、计费周期、费用项目与房间关联。不同合同不同定价 | 5-7天 |
| 3 | **房间批量调价** | 批量修改多个房间的费率，而不是逐个编辑 | 1-2天 |
| 4 | **按季度/年度出账** | 账单生成逻辑需要支持非月度的计费周期 | 2-3天 |
| 5 | **登录 + 权限系统** | 无权限控制无法给员工使用。操作员/管理员/只读等角色 | 3-5天 |

### P1 — 重要增强

| # | 缺失功能 | 说明 | 估算工作量 |
|---|---------|------|-----------|
| 6 | **业主自助缴费（微信/小程序）** | 业主在线查账、缴费。需对接微信支付或支付宝 | 10-15天（含支付对接） |
| 7 | **金蝶/用友导出模板** | 按金蝶要求的 Excel/CSV 格式导出财务数据 | 1-2天 |
| 8 | **车辆/停车位管理** | 车位与房间关联，独立的停车费计费 | 2-3天 |
| 9 | **押金管理** | 收退押金记录，与账单分离 | 1-2天 |
| 10 | **短信/消息通知** | 催缴通知通过短信发送，而非仅纸质打印 | 3-5天（含短信服务对接） |

### P2 — 扩展方向

| # | 功能 | 说明 |
|---|------|------|
| 11 | REST API | 暴露 API 供第三方系统调用 |
| 12 | 仪表盘可视化 | 接入 Chart.js 等图表库增强数据分析 |
| 13 | 多项目管理 | 一个系统管理多个物业小区 |

## 三、推荐迭代路线

### 第一阶段：商业版 MVP（当前→6周）

解决商业物业核心痛点，聚焦 P0：

```
第1-2周: 登录 + 权限系统
         └─ Admin/操作员账号、页面权限、会话管理

第3-4周: 合同驱动计费 + 批量调价
         └─ 合同模板、费率关联房间合同、批量更新

第5周: 公摊费用分摊引擎
     └─ 按面积比例 / 按用量比例 / 按户数三种模式

第6周: 季度/年度出账支持 + 全面测试
     └─ 账单生成适配 billing_cycle、回归测试
```

### 第二阶段：运营增强（7-12周）

```
第7-8周:  业主自助缴费（生成缴费二维码、微信小程序/H5）
第9-10周: 短信通知（催缴、缴费确认、报修进度）
第11周:   金蝶财务导出模板
第12周:   停车场管理 + 押金管理
```

### 第三阶段：平台化（13周+）

```
REST API → 电子发票 → 多项目管理 → 移动端
```

## 四、技术架构建议

如果往商业版走，建议适时做以下技术决策：

### 数据库
- SQLite 在**单项目、<10万条账单**的场景下够用
- 如果管理多个小区或数据量增大，建议迁移到 **SQLite 继续用**（省运维）或 **PostgreSQL**（需部署）

### 架构
- 当前"零依赖"策略的优势是部署简单，劣势是功能受限
- 如果要对接金蝶/短信/支付，**不可避免需要引入第三方库**（requests, openpyxl 等）
- 建议保留 server.py 作为轻量核心，新功能用独立模块引入

### 前端
- 当前服务端渲染在功能迭代阶段效率最高
- 如果需要业主自助端（小程序/H5），建议独立开发前端项目，通过 API 对接

### 金蝶对接
- **最小成本方案**：导出 CSV 由财务人工导入金蝶（0 开发，已有 CSV 可适配）
- **标准方案**：生成金蝶标准格式的 Excel，用 openpyxl 写模板（约 2 天开发）
- **深度对接**：通过金蝶 API 直接推送凭证（需金蝶开放平台支持，周期不定）

## 五、关于费用的问题

当前系统是**自建工具**，不是商业软件产品。如果未来要：

- **对外销售/授权**：需要重构为商业化产品，定价参考：按项目 + 按房间数梯度的 SaaS 模式
- **自己继续深化**：没有额外费用，只有开发时间投入
- **找人做二次开发**：按功能点报价，建议找熟悉 Python + SQLite 的开发者

---

## 六、总结建议

1. **当前系统定位**：小型住宅物业管理够用，商业物业缺关键能力
2. **最小投入拿结果**：先补 P0（登录+合同计费+公摊分摊），其他按需迭代
3. **不要一次做完**：建议按"先能用→再好用→再扩展"的顺序推进
4. **数据安全**：SQLite 单文件容易备份，建议每天自动备份到云盘
