# 真实数据试运行保护方案

## 一、当前阶段判断

从下一步开始，用户录入的是**真实业务数据**，不再按模拟数据处理。

因此当前阶段目标不是继续加新功能，而是保证：

- 真实数据不被误删。
- 真实数据不被测试数据覆盖。
- 真实数据不被提交到 GitHub。
- 真实账单金额能被复核。
- macOS / Windows 本机数据能备份、能恢复、能定位。

## 二、真实数据和试用数据的边界

| 数据类型 | 用途 | 可否清空 | 处理原则 |
|---|---|---|---|
| 试用数据 | 演示、测试、客户体验 | 可以，但必须先备份 | 可使用清空试用数据脚本 |
| 真实数据 | 真实房间、业主、收费、账单、收款 | 不允许随意清空 | 必须先备份、再操作、再验收 |
| GitHub 代码 | 程序源码和文档 | 不包含业务数据 | 严禁提交数据库和密钥 |

一旦开始录入真实数据，以下脚本只允许在明确确认“这台电脑只是试用环境”时使用：

```text
清空本机试用数据.command
清空本机试用数据.bat
```

## 三、真实数据保存位置

macOS：

```text
~/Library/Application Support/PropertyFeeSystemData
```

Windows：

```text
%APPDATA%\PropertyFeeSystemData
```

主要文件：

| 文件/目录 | 说明 |
|---|---|
| `property.db` | 真实业务数据库 |
| `backups/` | 自动备份和手动备份 |
| `startup_error.log` | 启动失败日志 |
| `runtime_status.log` | 诊断信息 |

## 四、真实数据导入前必须做

1. 确认当前电脑是否已经有旧数据。
2. 进入「数据备份」页面，手动创建备份。
3. 记录当前备份文件名和时间。
4. 确认导入文件来源可靠。
5. 核对导入 Excel 中的关键字段：
   - 楼栋
   - 单元
   - 房号
   - 楼层
   - 面积
   - 业主姓名
   - 手机号
   - 房屋类型
   - 合同起止时间
6. 不确定的数据先不要导入，先标记出来让业务人员确认。

## 五、真实数据导入后必须核对

导入后先不要急着生成账单，先核对：

| 核对项 | 核对方法 | 通过标准 |
|---|---|---|
| 房间数量 | 房间管理列表 / 导入源表 | 数量一致 |
| 业主数量 | 业主管理列表 / 导入源表 | 数量合理，重复业主可解释 |
| 面积合计 | 抽样或导出后汇总 | 与源表差异可解释 |
| 房号 | 抽查每栋每层 | 无明显错位 |
| 手机号 | 抽查业主端登录手机号 | 能匹配业主 |
| 房屋类型 | 居民/商户/特行等 | 与收费标准匹配 |
| 收费项目 | 收费项目页面 | 单价、计费方式正确 |

发现数量、面积、金额差异较大时，必须暂停，不要继续生成账单。

## 六、生成真实账单前必须做

1. 再次手动备份。
2. 先使用预览模式。
3. 抽查不少于以下样本：
   - 普通居民房间 3 户
   - 商户房间 3 户
   - 高楼层电梯费房间 3 户
   - 有特殊水费/特行收费的房间 3 户
4. 核对每类费用计算公式。
5. 金额确认后再正式生成。

## 七、真实收款前必须做

真实收款前要确认：

- 账单金额正确。
- 收款方式正确。
- 操作员账号正确。
- 收据编号规则正确。
- 对账报表能查到收款。

如果只是测试流程，不要使用真实业主数据做随意收款。可以单独建一套“测试房间/测试业主”，但必须能和真实数据区分开。

## 八、真实数据不进入 GitHub

严禁提交以下内容：

```text
.env
property.db
*.db
property.db-shm
property.db-wal
backups/
*.log
```

当前仓库已经通过 `.gitignore` 排除这些文件，但真实数据开始后，每次提交前仍要检查：

```bash
git status --short --ignored
```

如果看到数据库、备份、密钥文件，不能提交。

## 九、在线支付接入前的真实数据要求

等 macOS 和 Windows 都完整跑通后，再接微信/支付宝在线支付。

接支付前，真实数据至少要完成一次闭环验收：

1. 导入真实房间/业主。
2. 生成真实账单。
3. 后台手工登记一笔真实或受控测试收款。
4. 对账报表金额正确。
5. 业主端能看到自己的账单。
6. 电子票据入口和申请流程能打开。
7. 数据备份能找到并可说明恢复方式。

## 十、误操作处理原则

如果误导入、误生成、误收款：

1. 立即停止继续操作。
2. 不要反复覆盖数据。
3. 记录发生时间和操作人。
4. 找到误操作前最近备份。
5. 先复制一份当前数据目录，再考虑恢复。
6. 恢复后重新核对房间、账单、收款、报表。
