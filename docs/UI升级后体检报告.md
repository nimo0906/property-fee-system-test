## 🔴 遗留问题 1：测试断言过时（小，必须修）

UI 升级改了页面文案/结构，但**测试没同步更新**，导致 **7 个测试文件、共 11 个测试失败**。全部是"测试还在找旧文案"，不是功能坏：

| 文件 | 失败数 | 典型原因 |
|------|--------|----------|
| int_27 | 4 | 期望"账期对账"，页面现为"对账报表" |
| int_28 | 2 | 期望"主要收费类型统计"等旧标题 |
| int_01/11/15/21/26 | 各 1 | 旧标题/旧标签文字（如"提前预览"→"刷新预览"） |

**关键判断**：所有失败页面 HTTP 状态码都是 200，没有任何页面崩溃。这些是 codex 改 UI 时漏了"改动用户可见行为要同步测试"这一步。

**修法**：把这 11 个测试的断言更新为新文案即可（后端可做，是测试维护）。这是必须做的——否则测试一直红着，将来真出 bug 时会被淹没在已知失败里，失去预警作用。

## 🟠 遗留问题 2：CSS 合并质量不达标（中，建议再治理）

文件数虽从 18 降到 3，但**内在污染反而加重了**：

- 总行数 **3476 → 4450**（+28%）
- `.sidebar` 选择器定义次数 **47 → 82**
- **`!important` 共 531 个**（每 8.4 行一个；app-components.css 里 356 个，每 5.4 行一个）
- 同文件内同选择器重复定义严重：`.workbench-hero` 在 overrides 里 5 次、`.sidebar` 在 components 里 6 次

**原因**：codex 做"物理合并"时**没做去重**（我的交接文档把去重列为"可选第二步"，codex 跳过了），且 UI 升级又叠加了多轮视觉调整（app-overrides.css 里有 9 段带日期的"polish/redesign"注释）。本质上是"把 18 个小补丁文件合并成 3 个大补丁文件，然后又在上面叠了新补丁"，靠大量 `!important` 互相强压。

**影响**：不影响功能和当前显示，但维护性差——改样式仍可能"动一处崩别处"。

**修法**（codex 执行，非紧急）：在 3 个文件内做真正的去重（同选择器多次定义合并为一次，保留最终生效值），并约束 `!important` 使用。可作为下一轮前端整理任务。

---

## 建议的收尾顺序

1. **先修测试断言**（11 处，必须）——让测试恢复全绿，重建预警能力。
2. **CSS 再去重**（codex，建议但非紧急）——把 531 个 !important 和重复定义清理掉，否则 CSS 维护性问题会持续。
3. 给 codex 的常驻约定（codex.md）补一条：**改动用户可见文案/结构后，必须同步更新对应测试**——从源头避免测试再次滞后。
