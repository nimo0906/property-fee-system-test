# 后续软件开发可复用经验记录

## 一、总体研发方法

本项目从一个本地物业收费系统逐步迭代到 macOS / Windows 双端桌面交付版，形成了一套以后开发其他业务软件也可以复用的方法。核心原则是：先把真实业务主流程跑通，再逐步做安全、权限、打包、交付和上线冻结，不要一开始就做成过大的平台系统。

## 二、需求确认经验

### 1. 先围绕真实业务流程，不围绕技术想象

本项目真正有效的需求不是“做一个物业系统”，而是拆成了清晰流程：房间/业主/租户资料、收费项目、抄表、出账、缴费、开票、催缴、报表、结账、备份、用户权限。以后开发其他系统，也应先画出用户每天真实操作顺序，再决定页面和数据结构。

### 2. 用户现场注释非常重要

浏览器注释暴露了很多纯测试发现不了的问题，例如：

- 页面看不懂。
- 按钮点了没反馈。
- 筛选默认值不符合工作习惯。
- 下拉内容太混乱。
- 应该按租户显示而不是按业主显示。

以后做系统时，应保留“边使用边标注”的反馈模式。真实用户指出的位置，比开发者自己猜测更准确。

### 3. 业务词要统一

本项目中“业主、租户、商户、房间、铺位、账期、服务期、缴费周期”容易混用。后期修正后，系统稳定性明显提升。以后开发任何业务系统，都要尽早确定业务词汇表，页面、数据库、导入模板、报表都按同一套词来表达。

## 三、架构可复用经验

### 1. 本地桌面软件可以采用“本地 Web 服务 + 桌面壳”模式

本项目采用 Python 本地服务 + HTML 页面 + PyInstaller 打包桌面 App。优点是：

- 页面开发快。
- 业务逻辑容易测试。
- macOS / Windows 可以复用同一套后端逻辑。
- 后续如需移动端或局域网访问，也有扩展空间。

这种模式适合内部管理系统、财务录入系统、仓库系统、门店系统等轻量本地软件。

### 2. 用户资料目录必须与程序目录分离

本项目最终统一到独立数据目录：

- macOS：`~/Library/Application Support/PropertyFeeSystemData/`
- Windows：`%APPDATA%\PropertyFeeSystemData\`

目录结构包含 database、backups、imports、exports、updates、logs。这个设计非常值得复用。以后任何桌面软件都应避免把数据库、导入文件、导出文件、日志放进程序目录，防止升级、重装、打包时混乱或误删。

### 3. 高风险操作必须先备份

“清空试用业务数据”“恢复备份”“系统更新”等操作都属于高风险。最终采用的模式是：

- 管理员权限。
- 明确页面说明。
- 输入确认词。
- 执行前自动备份。
- 不删除程序文件、备份文件、导出文件。

以后其他系统也应沿用这个模式，尤其是财务、库存、订单、客户资料系统。

## 四、安全和权限经验

### 1. 权限不要只控制菜单，也要控制直连路由

本项目多次检查了隐藏模块、业主端、支付订单等入口。经验是：隐藏菜单不等于安全，必须保证直接输入 URL 也不能访问未上线模块。

以后系统中任何“暂时不上线”的模块，都应做到：

- 菜单不显示。
- 直连返回 404 或无权限。
- 数据表可保留，但入口停用。

### 2. 管理员账号要分层

本项目后期区分了超级管理员、业务管理员、财务收费、客服只读等角色。这个经验可复用：

- 开发/维护人员不应和客户日常管理员混用同一个最高权限账号。
- 客户日常使用应给业务管理员或财务角色。
- 系统更新、备份恢复、用户管理等高风险功能应保留更高权限。

### 3. 默认密码必须提醒但不强制联网注册

本项目是本地版，不需要互联网注册账号，但正式使用必须提醒默认管理员密码风险。以后做本地软件，也可以采用：默认可用、明显提醒、引导修改，而不是强制用户走复杂注册流程。

## 五、测试和验证经验

### 1. 每个用户反馈都尽量补成测试

本项目中很多问题最后都变成了集成测试，例如账期默认全部、发票待开票账单、隐藏模块 404、清空前备份、角色权限、桌面资源检查。以后开发时，凡是用户明确指出的问题，最好补一条测试，防止后面又修坏。

### 2. 上线前必须有固定验证命令

本项目最终固定了三类验证：

```text
PYTHONPYCACHEPREFIX=/private/tmp/property_pycache python3 -m py_compile server/*.py server.py desktop_app.py desktop_runtime.py
PYTHONPYCACHEPREFIX=/private/tmp/property_pycache python3 -m pytest -q
python3 scripts/desktop_release_check.py
```

以后其他项目也应建立类似的“上线前必跑命令”，不要靠人工感觉判断。

### 3. 桌面 App 要做真实启动验证

仅网页能打开不代表桌面包可用。本项目最终加入了 macOS App 打包、解压、符号链接、可执行文件和 serve-only smoke check。以后只要是桌面软件，必须验证打包后的程序，而不是只验证源码运行。

## 六、打包交付经验

### 1. macOS `.app` 不能随便用普通 zip 打包

本项目遇到过 zip 解压后 App 点不开的问题，原因是 `.app` 内部符号链接被破坏。最终改用：

```text
ditto -c -k --sequesterRsrc --keepParent
```

以后打包 macOS App 时，应优先使用 `ditto`，并解压后检查可执行文件和 symlink。

### 2. Windows 必须在 Windows 上打包

PyInstaller 不能从 macOS 可靠交叉生成 Windows exe。本项目采用：

- macOS 上准备 Windows build-source zip。
- Windows 电脑运行 `package_windows_release.bat`。
- 生成正式 `物业管理收费系统-v2.0-windows.zip`。

以后做双端桌面软件，也应把“源码准备包”和“正式 exe 包”区分清楚。

### 3. 交付包不能包含真实数据

本项目每次 zip 都检查不包含：

- `.env`
- `property.db`
- `backups/`
- `真实数据/`
- 缓存、日志、构建临时目录

以后所有项目都应建立类似检查，尤其是客户数据、合同、财务、账号信息不能进入 GitHub 或交付包。

## 七、上线冻结经验

### 1. 功能冻结非常重要

本项目后期功能已满足需求后，进入 v2.0.2 冻结版。冻结后只接受 P0/P1：

- 无法启动。
- 金额错误。
- 账期错误。
- 数据丢失。
- 越权访问。
- 主流程阻塞。
- 打包不可用。

以后其他系统上线前也应设置冻结点，避免上线前不断加功能导致新 bug。

### 2. P2/P3 先记录，不阻塞上线

发票真实税务平台对接、深度报表、自动更新增强、Windows 签名安装包等，都是有价值但不阻塞本次上线的优化。以后也应区分：必须上线前解决的问题，和上线后迭代的问题。

## 八、UI 和交互经验

### 1. 页面要让用户知道“现在看到的是什么”

本项目中结账、报表、发票、自动出账页面都曾出现“用户看不懂”的情况。后续通过标题、说明、筛选提示、空状态、分组展示改善。以后做系统页面时，每个页面至少要回答：

- 这个页面干什么？
- 当前显示的是全部还是筛选结果？
- 用户下一步应该点哪里？
- 空数据时为什么为空？

### 2. 下拉框数据多时必须分类

本项目中收费项目、房间、待开票账单都出现过混在一起难选择的问题。后续用单元/区域二级选择、商户分组、物业/商业/其他分类解决。以后只要下拉内容超过几十项，就应考虑分组、搜索或二级选择。

### 3. 默认筛选要符合用户习惯

本项目多次修正“默认某个月”改为“默认全部，用户手动选择才筛选”。经验是：默认值不要替用户缩小范围，除非业务上明确需要。

## 九、代码管理经验

### 1. 不要把交付散文件长期放仓库

本项目最后删除了旧的解包 release 目录，只保留最新版 zip，减少 GitHub 混乱。以后 release 目录应尽量只保留明确交付物，不保留重复解压目录、临时包、旧版本散文件。

### 2. 大功能完成后要整理代码和冻结记录

本项目在上线前做了：

- 旧目录清理。
- 无用模块治理。
- release zip 同步。
- 冻结文档。
- tag 标记。

以后项目也应在上线前做一次“代码瘦身 + 交付目录整理 + 冻结记录”。

### 3. GitHub 只放可公开、可复现、可交付的内容

本项目最终保留源码、测试、文档、最新版交付 zip；不上传本机数据库、备份、真实数据。这是以后所有项目都应遵守的基本线。

## 十、可复用检查清单

以后开发其他系统，可以直接按这个顺序检查：

1. 真实业务主流程是否跑通。
2. 用户资料目录是否和程序目录分离。
3. 高风险操作是否有权限、确认、备份。
4. 默认账号是否有安全提醒。
5. 隐藏模块直连是否不可访问。
6. 角色菜单和实际权限是否一致。
7. 默认筛选是否符合用户习惯。
8. 关键用户反馈是否补了测试。
9. 打包后的 App/Exe 是否真实启动过。
10. 交付包是否不含真实数据、`.env`、数据库、备份。
11. macOS zip 是否使用 `ditto`。
12. Windows exe 是否在 Windows 机器打包验证。
13. GitHub 是否只保留源码、文档、测试和最新版交付包。
14. 上线前是否有冻结 commit/tag 和冻结说明。
15. 冻结后是否只修 P0/P1。

## 十一、下一次项目建议直接复用的项目模板

如果以后要做新的内部管理软件，建议一开始就建立这些基础文件和目录：

```text
server/                  后端业务模块
static/                  前端样式和脚本
templates/               页面模板
tests/                   自动化测试
scripts/                 发布检查和辅助脚本
docs/                    交付文档、冻结记录、上线清单
release/macos/           macOS 交付包
release/windows/         Windows 交付包
```

并预置这些能力：

- 统一数据目录。
- 自动备份。
- 用户权限。
- 默认密码提醒。
- 桌面打包脚本。
- 发布前检查脚本。
- 测试命令。
- release zip 安全检查。
- 冻结记录文档。

## 十二、最重要的结论

本项目最有复用价值的不是某一段代码，而是一套交付方法：

```text
真实业务流程优先 → 小步迭代 → 用户现场反馈 → 测试固化 → 桌面打包验证 → 数据安全隔离 → 版本冻结 → GitHub 清理同步 → 上线后只修高优先级问题
```

以后做其他软件，只要沿用这套方法，就能减少返工、减少上线风险，也更容易把项目从“能跑”推进到“能交付、能维护、能给真实用户用”。

## 十三、本次商场收费和 Windows 服务器共享部署新增经验

### 1. 复杂业务不要一开始做成“大合同系统”

商场商业收费最初的问题是面积、单价、缴费周期都不统一。如果直接做完整合同系统，会把项目复杂度拉得过高。最终更实用的方案是：

- 先做“商户档案 + 收费规则”。
- 商户资料里维护铺位号、店铺名称、业态、面积、物业费单价、缴费周期、合同起止日期、水费标准。
- 出账时按商户自己的面积、单价、缴费周期计算。
- 面积采用直接覆盖，新账单按新面积，历史账单不回改。

这个经验适用于以后开发其他业务软件：遇到合同、规则、审批、流程很多的业务时，先抽出当前真实使用所需的最小档案和规则，不要一上来做全平台。

### 2. 收费、库存、订单类系统必须先定义“命中范围”

本次商业收费明确了：

- A 座独立运行，不进入系统收费范围。
- B 座住宅按原住宅收费逻辑，不被商业收费命中。
- 商业收费只纳入 `unit=商场` 且 `category=商户/商业` 的对象。

这类“范围隔离”比公式本身还重要。以后做库存、会员、订单、租赁、合同等系统，也要先回答：这条规则到底命中哪些对象，不命中哪些对象。否则很容易出现算对了公式，但用错了对象。

### 3. 多人共用数据时，不要共享 SQLite 文件，要共享服务入口

本项目最终采用 Windows 服务器共享模式：

- 服务器上只运行一个系统服务。
- 数据库放在服务器固定数据目录。
- 操作员电脑通过浏览器访问服务器地址。

这是正确做法。不要让多台电脑各自运行程序后同时打开同一个 SQLite 数据库文件，否则容易锁库、冲突或损坏数据。

以后本地业务系统如果需要多人同时使用，优先采用：

```text
一台服务器运行服务 → 所有人访问同一个 HTTP 地址 → 数据只由服务器进程读写
```

### 4. Windows 交付必须区分“源码部署包”和“EXE 免环境包”

本次现场部署暴露了一个关键问题：源码部署包需要 Python，而旧 Windows Server 安装 Python 可能失败。后续更稳的交付方式是：

- 普通使用者优先发 EXE 免 Python 包。
- 服务器部署也优先发 EXE 免 Python 包。
- 源码部署包只给开发、维护或明确会配置环境的人使用。
- 包名必须写清楚：普通桌面、服务器共享、源码版、EXE 免 Python 版。

以后所有 Windows 桌面/内网业务系统，都要默认优先交付 EXE 包，不要让客户现场先装 Python、Node 或其他运行环境。

### 5. 服务器共享模式的现场操作说明必须写到“可照抄”

本次真正有效的说明是：

```text
服务器黑窗口不要关。
服务器本机可用 http://127.0.0.1:5001 测试。
其他电脑用 http://服务器IPv4:5001。
SERVER_IP 不是原样复制，要替换成真实 IPv4。
如果打不开，放行防火墙 5001 端口。
```

以后写部署说明时，不能只写“访问服务器地址”，必须说明：

- 黑窗口/服务进程是否要保持运行。
- 本机地址和其他电脑地址的区别。
- 如何用 `ipconfig` 找 IPv4。
- 浏览器要求。
- 防火墙放行命令。
- 服务器不能关机、不能休眠。

### 6. 不要默认服务器自带浏览器可用

这次 Windows Server 自带 Internet Explorer 能打开页面，但样式显示混乱。原因不是系统坏了，而是 IE 太旧，不支持现代页面样式。

以后所有本地 Web/桌面壳系统交付时，应明确：

- 推荐 Chrome、Edge、Firefox、360 极速模式。
- 不支持 Internet Explorer。
- 如果页面乱，先换浏览器，再排查静态资源。

### 7. 离线/内网系统不能依赖外部 CDN

本项目已经把 Bootstrap、Bootstrap Icons、Chart.js 等静态资源本地化。这个经验必须复用：

- 内网服务器可能不能访问外网。
- 客户现场网络可能拦截 CDN。
- 页面样式和图表不能依赖公网资源。

以后任何本地部署系统，前端依赖都应打进 `static/vendor/`，并在打包检查里确认这些资源被包含。

### 8. 移动端和真实支付要按阶段做，不能只加按钮

业主移动端可以后续接微信/支付宝，但真实支付不是前端按钮问题。至少需要：

- 支付订单。
- 微信/支付宝签名下单。
- HTTPS 公网回调地址。
- 异步回调验签。
- 幂等入账。
- 对账、退款、关闭订单。

所以后续其他软件如果要接支付、短信、发票、银行、企业微信等外部平台，也要先做内部订单/状态闭环，再接真实平台。
