# SaaS 云端商业版下一阶段立项决策清单

本文用于上线后 7 日稳定运行复盘通过后，决定是否进入下一阶段。当前第一版范围仍是员工后台商业版；下一阶段必须单独立项、单独验收，不得与现有 SaaS 业务主库混用，不得破坏租户数据隔离，不得把客户上传数据与系统自身数据混在一起。

## 立项前置条件

- `docs/saas-day7-stability-review.md` 已完成并通过。
- `release/saas-commercial-launch-report.pdf` 已留档。
- `scripts/saas_release_gate.py` 仍然通过。
- 7 日内无 P0 级金额、租户隔离、权限、备份恢复、审计问题。
- 客户反馈不影响当前员工后台收费业务。

## 候选方向

| 方向 | 说明 | 是否建议立即做 |
| --- | --- | --- |
| 业主端 H5 | 面向业主移动端查询账单、缴费入口、通知和个人信息。 | 7 日稳定后可评估 |
| 微信/支付宝真实支付 | 支付订单、签名回调、幂等、退款、对账和 HTTPS 公网部署。 | 需支付专项设计后再做 |
| 授权云服务后台 | 商业授权、客户版本、授权状态、到期提醒，与 SaaS 业务后台分离。 | 独立后台后置 |
| 更多桌面版存量业务迁移 | 商户合同、水电表、停车费、公摊、更多报表。 | 按客户付费需求排期 |

## 优先级建议

1. 若客户已稳定使用员工后台并强需求业主自助查询，优先立项业主端 H5。
2. 若客户明确要线上收款，先做支付专项方案，再立项微信/支付宝真实支付。
3. 若开始批量售卖给多家公司，再立项授权云服务后台。
4. 若客户付费场景集中在商业综合体或停车/水电，优先迁移对应桌面版存量业务。

## 进入条件

- 当前员工后台无 P0 遗留问题。
- 租户数据隔离和客户上传数据与系统自身数据隔离已验证。
- 备份恢复演练可重复执行。
- 客户确认下一阶段有明确业务价值和预算。
- 每个方向都有独立验收标准、测试计划、回退方案。

## 不进入条件

- 当前版本存在租户越权、金额异常、审计缺失、备份失败。
- 客户上传数据与系统自身数据隔离未通过。
- 还没有真实客户稳定使用证据。
- 只是为了“功能看起来完整”而没有付费业务需求。

## 风险和边界

- 不得与 SaaS 业务主库混用：授权云服务后台应作为独立后台或独立 schema/service 设计。
- 不得破坏租户数据隔离：业主端 H5 和支付回调也必须严格带租户和项目范围。
- 不得把客户上传数据与系统自身数据混在一起：业主端附件、支付对账文件、授权文件必须分目录保存。
- 微信/支付宝真实支付不能只做按钮，必须包含支付订单、回调验签、幂等、退款/取消、对账和异常补偿。
- 业主端 H5 不应反向扩大员工后台权限。
