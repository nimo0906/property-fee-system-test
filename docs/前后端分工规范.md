# 前后端分工规范（Claude 后端 / Codex 前端）

> 目的：解决「改一个页面、其它同类页面没跟着改」「修好这里那里坏」的问题。
> 制定日期：2026-06-25　适用项目：物业管理收费系统（测试版本）

---

## 一、为什么要分工（问题诊断）

对当前代码扫描后发现的事实：

- 全系统约 **75000 行、559 个 py 文件**（一个月前是 19566 行、158 文件）。
- HTML 按钮 **90% 以上写死在 Python 里**：py 文件内有 136 个 `<button>`，模板里只有 14 个。
- 「打印」逻辑散落在 **19 个文件、24 处**；已有统一组件 `print_helper.py`，却只有 6 处用到它，其余各写各的。
- `static/` 下有 **20 多个 CSS 文件**（`app-command-1-1.css`、`app-polish.css`、`app-redesign-1.css`…），是反复「打补丁盖问题」的痕迹。

**结论**：地基其实是好的——`_load_template()` / `render_page()` / `print_helper.py` 这套统一渲染机制已经存在，已有 9 个页面在用模板。问题不是没有正确做法，而是**两套做法并存、没有规矩强制收敛**，于是谁都只改眼前那一处。

---

## 二、核心边界（一句话）

> **后端（Claude）只产出数据和业务结果，绝不写 HTML 样子。**
> **前端（Codex）只决定页面长什么样，绝不碰计费和数据库逻辑。**

两边的交界面是：**模板文件（`templates/*.html`）** 和 **统一组件（`print_helper.py` 等）**。

## 四、协作约定（最重要的几条）

### 约定 1：同类功能必须走统一组件，禁止复制

凡是「每个页面都有」的东西——**打印、返回按钮、导出、分页、表格、表单**——必须使用已有的统一组件，不允许在新页面里手写一份。

- 打印 → 一律 `from server.print_helper import print_page`，禁止自己拼 `window.print()`。
- 页面外框 / 导航 / 返回 → 一律 `render_page()`，禁止自己写 `<html><head>`。
- 分页 → 一律 `server/pagination.py` 的 `render_pagination()`。

**判断标准**：如果一个 UI 元素在两个以上页面出现，它就必须有唯一的一份实现。Codex 改它时改那一份，全局自动生效——这正是「全局维修」的实现方式。

### 约定 2：HTML 逐步搬出 Python，搬进模板

现状是 76 个 py 文件里拼着 HTML。**不要求一次搬完**，规则是：

- **新功能**：HTML 一律写在 `templates/` 里，Python 只准备数据。
- **改老功能**：谁改到哪个页面，就顺手把那个页面的 HTML 从 py 搬进模板（"路过就修"）。
- 搬迁时**只搬 HTML 结构，不动数据逻辑**——数据逻辑留在 py，由 Claude 负责。

### 约定 3：改动前先问「全局有几处」

这是针对你遇到的核心痛点。**任何一方动手改一个功能前，先全局搜一遍**：

```
# 例：要改打印，先搜全局有多少处打印
grep -rln "打印\|window.print" server/
```

- 如果只有一处（已收敛到组件）→ 直接改，全局生效。
- 如果有多处（还没收敛）→ **先收敛成一处再改**，不要只改眼前这个。

### 约定 4：禁止「新加 CSS 文件盖问题」

样式有问题，改源头的那个类，不允许再新建 `app-fix-N.css` 去覆盖。
现有的 20 多个 CSS 文件需要逐步合并——这是 Codex 的一项专门任务（见第六节）。

### 约定 5：交界面变更要互相知会

- Claude 改了传给模板的数据字段名 → 必须在交接说明里写明，让 Codex 同步模板。
- Codex 发现模板需要某个新数据 → 提需求给 Claude，不要自己去 py 里查数据库。

## 五、一次改动的标准流程

```
用户提需求
  ↓
判断：是数据/计算问题，还是页面样子问题？
  ├─ 数据/计算 → Claude 改 server 逻辑，产出/修正数据
  └─ 页面样子 → Codex 改 templates / static
  ↓
改动前：grep 全局，确认这个功能有几处实现
  ├─ 一处 → 直接改
  └─ 多处 → 先收敛到统一组件，再改
  ↓
验证：py_compile + pytest（分文件跑，见测试隔离说明）
  ↓
说清楚改了哪些用户可见行为
```

## 六、建议的推进顺序（从你的痛点出发）

1. **先收敛「打印」**（你举的例子）：把 19 处打印全部改成调用 `print_helper.print_page`，删掉各自手写的版本。完成后，以后改打印样式只动 `print_helper.py` 一处。——这是验证整套分工能跑通的试点。
2. **再收敛「按钮 / 表格 / 表单」**等其它高频重复元素。
3. **合并 CSS**：把 20 多个 `app-*.css` 理成一份主样式，去掉互相覆盖。
4. **逐步把 76 个 py 里的 HTML 搬进模板**（路过就修，不搞大爆炸式重写）。

## 七、给 Codex 的补充约束（追加到 codex.md）

- 前端只动 `templates/`、`static/`，以及 py 文件里**纯 HTML 字符串**部分；不动任何 `def` 里的查询/计算/金额逻辑。
- 改任何「每页都有」的 UI 元素前，先 `grep` 全局确认实现处数，多处先收敛再改。
- 不再新增 `app-*.css` 补丁文件；样式问题改源头类。
- 不动 `billing_*`、`bill_generation*`、`*_service.py`、`db.py`、数据库 schema。
