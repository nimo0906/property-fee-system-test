目标：沿用今天已确定的“正式提案版”方法，先给每个模块找到组件与成熟 UI 模式，再把它们落成可确认的 HTML 小样。确认后再改真实页面。

1. 这次实际用到的组件网址和 UI 网址

小样里的组件结构全部来自这些参考方向，再套回本系统的桌面业务密度。

组件库参考

  • Ant Design Table：长列表、筛选、排序、固定列。
  • Ant Design ProTable：查询表单 + 表格一体。
  • Ant Design Form：录入、校验、表单布局。
  • Ant Design Upload：导入文件和进度。
  • Ant Design Steps：导入、结账、备份流程。
  • Ant Design Drawer：详情侧滑，不离开列表上下文。

21st.dev 参考

  • Dashboard Components：经营看板和指标卡。
  • Table Components：密集表格样式。
  • Flexi Filter Table：筛选条 + 数据表。
  • Fixed Header Footer Table：长表固定表头/汇总。
  • Admin Panel Components：后台管理模块布局。

成熟 UI 模式参考

  • Pageflows Dashboard：看板信息优先级。
  • Pageflows Table：列表页操作密度。
  • Pageflows Search Bar：搜索与筛选入口。
  • Mobbin Finance Web：财务类 Web 页面。
  • Mobbin Drawer：详情和编辑侧栏。

2. 全菜单模块组件地图

首页、账单、合同已进入真实页面改造；下面是其他模块的提案方向。

物业收费

FormDrawer

收费对象、同租户房间、待收项目集中到一个收银台。

复刻：AntD Form + Drawer 详情。

商业收费

TabsTable

先选商户/空间，再看租金、物业费、水电费。

复刻：ProTable + Segment Tabs。

智能预警

TimelineBadge

合同到期、欠费、异常金额按风险等级聚合。

复刻：Timeline + Status Card。

催缴管理

KanbanDrawer

催缴对象按未联系、已联系、承诺付款分栏。

复刻：CRM pipeline + drawer。

数据导入

UploadSteps

上传、预览、问题行、确认导入四步清楚分离。

复刻：Upload.Dragger + Steps。

业主管理

TableDescriptions

左表右详情，业主、手机号、名下房间一眼可查。

复刻：Table + Detail Drawer。

房间管理

Tree/SegmentTable

B座/商场先分区，房号、面积、合同状态进入主列。

复刻：Segmented + Filter Table。

收费项目

Editable TableTag

项目启用、适用范围、排序、单价规则在一张表内管理。

复刻：Editable cells + Tags。

抄表管理

StatisticTable

本期待抄、已抄、异常用量、待确认固定在顶部。

复刻：Metric cards + Input table。

批量更新

StepsConfirm

选择范围、字段、预览影响、确认提交四步化。

复刻：Wizard + Diff table。

自动出账

ConsoleRun history

规则、预览、执行、历史记录做成控制台。

复刻：Admin console + timeline。

公摊分摊

Formula CardTable

分摊规则、对象范围、金额试算、生成账单连续展示。

复刻：Formula panel + result table。

缴费记录

LedgerDrawer

收款流水、支付方式、收据、冲红入口靠右固定。

复刻：Fixed footer table。

客服催费对象

ListTimeline

对象清单 + 跟进记录时间线。

复刻：Contact list + timeline。

发票管理

TabsStatus

待开票、已开票、作废三个稳定分区。

复刻：Tabs + invoice table。

对账报表

ChartsExport

摘要、趋势、结构、明细四块，不再一页只有表。

复刻：Finance dashboard。

期末结账

LockChecklist

结账前检查、差异、确认锁账分步。

复刻：Steps + checklist。

系统维护

AuditBackup

日志、备份、用户、健康、更新合并成维护控制台。

复刻：Settings dashboard。

7. 建议确认后的实施批次

确认这个小样方向后，再逐页改代码，仍按今天的方式：先测、再开真实页面核对。

第一批：高频操作

物业收费、商业收费、数据导入、业主管理、房间管理。

原因：员工每天用，改完感知最强。

第二批：财务核对

缴费记录、发票管理、对账报表、期末结账。

原因：金额、状态、追溯链要统一。

第三批：规则与自动化

收费项目、抄表管理、批量更新、自动出账、公摊分摊。

原因：需要更多业务边界核对。

第四批：系统维护

操作日志、数据备份、用户、云端技术备查、健康、更新、清空试用数据。

原因：更偏安全和桌面交付。
需要你确认：如果这个小样方向认可，我下一步先从“第一批：高频操作”开始，把真实页面逐个改成一致组件结构；每改完一页都跑测试并打开页面给你看。
